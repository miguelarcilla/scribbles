> This document is the authoritative specification for the Taskboard application.
> All architecture decisions, feature implementations, and infrastructure choices must align with this constitution.
> Changes to this document require explicit review and must be version-tagged.

---

## 1. Vision & Purpose

**Taskboard** is a multi-tenant, cloud-native shared to-do list web service. It enables organizations and their members to collaboratively manage tasks across boards in a secure, isolated, and scalable SaaS environment.

### Core Principles

1. **Tenant Isolation** — Every piece of data is scoped to a tenant. No data leakage between tenants is acceptable under any circumstance.
2. **Identity-First** — Authentication and authorization are not optional layers; they are foundational to every request.
3. **Cloud-Native** — The application is designed to run on Azure and leverages managed services wherever possible.
4. **Spec-Driven** — Features are defined by specifications before code is written. This constitution is the root spec.
5. **Simplicity Over Cleverness** — Prefer readable, maintainable code over complex abstractions.

## 4. Multi-Tenancy Model

### Tenant Definition

A **tenant** corresponds to a Microsoft Entra ID tenant (directory). When a user from an organization authenticates for the first time, their organization's tenant is automatically provisioned in Taskboard.

### Tenant Isolation Strategy

- **Data isolation**: All CosmosDB containers use `tenantId` as the partition key.
- **API isolation**: Every API request extracts `tid` (tenant ID) from the validated JWT. Controllers never accept a `tenantId` from the request body or query string—it is always derived from the token.
- **Cross-tenant access**: Not supported. Users can only access data within their own tenant.

### Tenant Lifecycle

```
User authenticates (first time from org)
    → API receives JWT with new tid
    → TenantService.ProvisionAsync(tid, tenantName)
    → Tenant document created in CosmosDB
    → User document created/updated
    → User proceeds normally
```

### User Roles (per tenant)

| Role | Description |
|------|-------------|
| `Owner` | Full control, can manage members and delete tenant data |
| `Admin` | Can manage boards and members, cannot delete tenant |
| `Member` | Can create/edit own tasks, view shared boards |
| `Viewer` | Read-only access to boards they are assigned to |

## 6. API Design

### Base URL

```
https://<api-hostname>/api/v1
```

### Versioning

URL-path versioning (`/api/v1/`, `/api/v2/`). Breaking changes require a new version.

### Auth

All endpoints require a Bearer token from Entra ID **except**:
- `GET /health` — health check (no auth)
- `GET /api/v1/status` — public API status

### Endpoint Inventory

#### Tenant
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/tenant` | Get current tenant info |
| PUT | `/api/v1/tenant` | Update tenant settings (Owner/Admin) |

#### Users
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/users` | List users in tenant |
| GET | `/api/v1/users/me` | Get current user profile |
| PUT | `/api/v1/users/me` | Update current user profile |
| PUT | `/api/v1/users/{userId}/role` | Change user role (Owner/Admin) |

#### Boards
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/boards` | List all accessible boards |
| POST | `/api/v1/boards` | Create a new board |
| GET | `/api/v1/boards/{boardId}` | Get board details |
| PUT | `/api/v1/boards/{boardId}` | Update board |
| DELETE | `/api/v1/boards/{boardId}` | Archive/delete board |
| GET | `/api/v1/boards/{boardId}/tasks` | List tasks on board |

#### Tasks
| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/v1/boards/{boardId}/tasks` | Create task |
| GET | `/api/v1/tasks/{taskId}` | Get task details |
| PUT | `/api/v1/tasks/{taskId}` | Update task |
| DELETE | `/api/v1/tasks/{taskId}` | Delete task |
| PATCH | `/api/v1/tasks/{taskId}/status` | Update task status |
| PATCH | `/api/v1/tasks/{taskId}/assign` | Assign task |

## 7. Authentication & Authorization

### Entra ID App Registration Requirements

Two app registrations are required:

**1. Taskboard API App Registration**
- Expose API: `api://<client-id>/access_as_user`
- Supported account types: **Accounts in any organizational directory (Multi-tenant)**
- No redirect URIs (API only)

**2. Taskboard Web App Registration**
- Supported account types: **Accounts in any organizational directory (Multi-tenant)**
- Redirect URI: `https://<static-web-app-url>/authentication/login-callback`
- API permissions: `api://<api-client-id>/access_as_user` (delegated)

### Token Validation (API)

The API validates:
- Signature (via OIDC discovery)
- `aud` (audience) — must match API client ID
- `iss` (issuer) — must match `https://login.microsoftonline.com/{tenantId}/v2.0`
- `tid` (tenant ID) — extracted and used for all data queries
- `oid` (object ID) — user's unique identifier within the tenant

### Frontend Auth Flow

1. User navigates to app → redirected to Entra ID login
2. Entra ID issues ID token + access token
3. Blazor stores tokens via MSAL
4. All API calls include `Authorization: Bearer <access-token>`
5. On expiry, MSAL silently refreshes via refresh token

```bash
# Deploy to a resource group
az deployment group create \
  --resource-group rg-taskboard-dev \
  --template-file infra/main.bicep \
  --parameters infra/parameters/dev.bicepparam
```

### Configuration Injection

App Service settings are injected by Bicep at deploy time:
- `AzureAd__TenantId` = `common` (multi-tenant)
- `AzureAd__ClientId` = API app registration client ID
- `CosmosDb__ConnectionString` = from Key Vault reference or Cosmos DB keys
- `CosmosDb__DatabaseName` = `taskboard`

## 11. Configurability

The database layer is abstracted via the `ITaskRepository` and `IBoardRepository` interfaces. While the default and recommended implementation is CosmosDB, alternative implementations (e.g., SQL Server via EF Core, MongoDB) can be registered in `Program.cs` without modifying service or controller code.

```csharp
// Default: CosmosDB
builder.Services.AddScoped<IBoardRepository, CosmosBoardRepository>();

// Alternative (swap in Program.cs):
// builder.Services.AddScoped<IBoardRepository, SqlBoardRepository>();
```
